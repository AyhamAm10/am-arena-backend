# AM Arena Backend — Architectural Contract (.cursor-role.md)

This file is the **permanent architectural contract** for the project. All code must conform to it. The Auth module is the **golden reference**; every module must follow the same structural and behavioral patterns.

---

## 1. Project structure (mandatory)

All application code lives under `src/`. No files may be created outside the following layout.

```
src/
├── index.ts                    # App entry: express, cors, cookieParser, langMiddleware, router mount, errorHandler last
├── config/                     # data_source, var, createSuperAdmin, etc.
├── entities/                   # TypeORM entities only (e.g. User.ts)
├── routes/                     # One file per module: <module>.route.ts
├── controllers/                # One file per module: <module>.controller.ts
├── services/
│   ├── repo.service.ts         # Base RepoService (all repo services extend this)
│   ├── repo/
│   │   └── <module>/           # One folder per domain module
│   │       └── <module>.service.ts
│   ├── jwt/
│   │   └── jwt.service.ts
│   ├── image.service.ts
│   └── queryBuilder.service.ts
├── dto/
│   └── <module>/               # One folder per module
│       └── <action>.dto.ts     # e.g. auth-register.dto.ts, auth-login.dto.ts
├── middlewares/                # auth.middleware, role.middleware, lang.middleware, etc.
├── common/
│   ├── errors/
│   │   ├── Ensure.handler.ts   # Ensure.* — only allowed way to throw domain/validation errors in services
│   │   ├── validator.ts        # validator(schema, data) for DTO validation
│   │   ├── ErrorMessages.ts    # generateErrorMessage(entity, errorType, lang)
│   │   ├── http.error.ts       # HttpError, BadRequestError, UnauthorizedError, ForbiddenError, NotFoundError, InternalServerError
│   │   ├── api.error.ts        # APIError, HttpStatusCode, BaseError
│   │   └── error.handler.ts    # Global error middleware
│   └── responses/
│       └── api.response.ts     # ApiResponse.success / ApiResponse.error
├── types/                      # e.g. express.d.ts
├── utils/
├── logging/
└── migrations/
```

**Rules:**

- No new top-level folders under `src/` other than those listed.
- No files outside the defined module structure (e.g. no loose controllers or services at `src/` root).
- Each **module** has exactly: one route file, one controller file, one service file under `services/repo/<module>/`, and one or more DTOs under `dto/<module>/`.

---

## 2. Module structure (mirror Auth)

Every module must mirror the Auth module layout and patterns.

| Layer        | Location                          | Naming / pattern                                      |
|-------------|------------------------------------|--------------------------------------------------------|
| Route       | `src/routes/<module>.route.ts`     | Default export: Express Router, use `<Module>Controller` |
| Controller  | `src/controllers/<module>.controller.ts` | Class `<Module>Controller`, methods bound to routes   |
| Service     | `src/services/repo/<module>/<module>.service.ts` | Class `<Module>Service extends RepoService<Entity>` |
| DTOs        | `src/dto/<module>/<action>.dto.ts` | One file per request shape (e.g. auth-register, auth-login) |

---

## 3. Controller pattern (strict)

- **Class-based**: one class per module, e.g. `AuthController`.
- **Methods**: `async (req: Request, res: Response, next: NextFunction)`.
- **No business logic**: controllers only validate input, call the service, and send the response.

**Required flow in each handler:**

1. `try {`
2. Get language: `const lang = (req.headers["accept-language"] as Language) || "en";`
3. **Validate body**: `await validator(<schema>, req.body);` then cast: `const dto = req.body as XxxDto;`
4. Instantiate service: `const service = new XxxService();`
5. Call service with DTO-derived data only (no business rules in controller).
6. Respond with `res.status(...).json(ApiResponse.success(data, ErrorMessages.generateErrorMessage(entity, type, lang)))` or `res.json(ApiResponse.success(...))`.
7. `} catch (err) { next(err); }`

**Response formatting:**

- Success: `ApiResponse.success(data, message, options?)`. Use `ErrorMessages.generateErrorMessage(entity, "created" | "retrieved" | "updated" | "deleted" | "logged in" | "logged out" | etc., lang)` for `message`.
- Created resources: `res.status(HttpStatusCode.CREATED).json(ApiResponse.success(...))`.
- Other successes: `res.json(ApiResponse.success(...))`.

**Controller error handling:**

- Do not implement business checks in the controller (e.g. “user exists”). Those belong in the service and use `Ensure`.
- For **request-context** checks only (e.g. `req.currentUser` missing after auth middleware), the controller may throw `APIError` with `HttpStatusCode` and a message from `ErrorMessages.generateErrorMessage(..., lang)`.
- All other errors must be delegated to the service or middleware and passed to `next(err)`.

---

## 4. Service pattern (strict)

- **Base class**: every domain service **must** extend `RepoService<Entity>` from `src/services/repo.service.ts`.
- **Constructor**: `super(Entity);` and, if methods are passed as callbacks, bind them: `this.methodName = this.methodName.bind(this);`
- **Location**: `src/services/repo/<module>/<module>.service.ts`.

**Responsibilities:**

- All business and domain logic.
- All data access for that domain via `this.repo` and RepoService methods: `create`, `update`, `delete`, `getById`, `getAll`, `getAllWithPagination`, `findOneByCondition`, `findManyByCondition`, `exists`, `count`, `createOrUpdate`, `createWithTransaction`, `createManyRaw`.
- **No** direct `AppDataSource.getRepository(...)` in service code; use only the repository provided by `RepoService` (`this.repo`).

**Error handling in services:**

- **Only** the `Ensure` class from `src/common/errors/Ensure.handler.ts` may be used to throw domain/validation errors.
- Use: `Ensure.exists(value, entity)`, `Ensure.alreadyExists(condition, entity)`, `Ensure.required(value, entity)`, `Ensure.unauthorized(condition, entity)`, `Ensure.forbidden(condition, entity)`, `Ensure.min(value, min, entity)`, `Ensure.isNumber(value, entity)`, `Ensure.isArray(value, entity)`, `Ensure.custom(condition, message)`.
- Do **not** throw raw `Error`, `APIError`, or `HttpError` in services; use `Ensure.*` (and `Ensure.custom` for custom messages).
- Do not catch and rethrow in services unless necessary; let `Ensure` and the global `errorHandler` handle responses.

---

## 5. DTOs and validation (mandatory)

- **Every API request that accepts body or query parameters must use a DTO** defined under `src/dto/<module>/`.
- **One file per request shape**: e.g. `auth-register.dto.ts`, `auth-login.dto.ts`, `update-profile.dto.ts`.
- **Content of each DTO file:**
  - A **yup** schema exported as `<name>Schema` (e.g. `authRegisterSchema`, `authLoginSchema`).
  - A TypeScript type exported as `<Name>Dto`: `export type XxxDto = yup.InferType<typeof xxxSchema>;`
- **Validation**: in the controller, before using request data, call `await validator(<schema>, req.body)` (or the appropriate request field). The `validator` is from `src/common/errors/validator.ts`; it throws `BadRequestError` on validation failure.
- **Usage**: after `validator(...)`, cast the request data to the DTO type (e.g. `req.body as AuthRegisterDto`) and pass only DTO-derived data to the service.

---

## 6. Error handling (strict)

- **In services**: use **only** the `Ensure` class. No `throw new Error(...)`, no `throw new APIError(...)`, no `throw new HttpError`/subclasses in service logic. Use `Ensure.custom(condition, message)` when you need a custom message (e.g. from `ErrorMessages.generateErrorMessage(...)`).
- **In controllers**: use `Ensure` only if you have a trivial guard that fits `Ensure`; for request-context failures (e.g. missing `currentUser`), you may throw `APIError` with `HttpStatusCode` and pass to `next(err)` in `catch`.
- **In middlewares**: pass errors to `next(...)` (e.g. `next(new UnauthorizedError(...))`). Middlewares may use `Ensure.exists(...)` for auth/identity checks.
- **Global handler**: `errorHandler` in `src/common/errors/error.handler.ts` must remain the last middleware. It handles `APIError`, `HttpError`, and generic `Error` and sends the correct status and JSON shape. No ad-hoc error response formatting in controllers or routes.

---

## 7. Repository and database access

- **Controllers and routes**: must **not** access the database or call `AppDataSource.getRepository(...)`.
- **Services**: all domain data access goes through **RepoService**: extend `RepoService<Entity>` and use `this.repo` and the base methods only. No direct `AppDataSource.getRepository(...)` in service files.
- **Middlewares**: may access the database **only** for authentication/authorization (e.g. loading the current user by id). This is the only exception to “no direct DB outside services.” Use `Ensure.exists(...)` after loading when a user is required.

---

## 8. Routing style

- **Per-module router**: `const router = Router(); const controller = new XxxController();`
- **Routes**: `router.<method>("<path>", handler)` or `router.<method>("<path>", middleware, handler)`.
- **Handler**: pass controller methods directly (e.g. `authController.register`). Ensure controller methods are bound in the constructor if they are passed as callbacks.
- **Export**: `export default router;`
- **Mount in index.ts**: `router.use("/<prefix>", moduleRouter)` (e.g. `router.use("/auth", authRouter)`). Base path is applied via `app.use(process.env.BASE_URL ?? "/", router)`.

---

## 9. Middleware usage

- **Order in index.ts**: `cors` → `express.json()` → `express.urlencoded` → `cookieParser()` → `langMiddleware` → routes → `errorHandler` (last).
- **Language**: use `getLanguage()` from `src/middlewares/lang.middleware.ts` (backed by AsyncLocalStorage). Controllers read `req.headers["accept-language"]` for response messages; services use `getLanguage()` when calling `ErrorMessages` or `Ensure`-related behavior that depends on lang.
- **Auth**: protected routes use `authMiddleware` (and optionally role/status middlewares) before the controller method. Auth middleware sets `req.currentUser` (user id) and uses `Ensure.exists(user, "user")` after loading the user.

---

## 10. Response and status codes

- Use `HttpStatusCode` from `src/common/errors/api.error.ts` for status codes (e.g. `HttpStatusCode.CREATED`, `HttpStatusCode.OK`).
- Success payload: always via `ApiResponse.success(data, message, options?)`. For pagination, pass `options` with `count`, `page`, `limit` so `meta` is set.
- Do not invent new response shapes; use the existing `ApiResponse` API only.

---

## 11. Prohibited deviations

- **No** business logic in controllers.
- **No** direct database or repository access in controllers or routes.
- **No** validation without a DTO and `validator(schema, ...)`.
- **No** throwing domain/validation errors in services without using `Ensure`.
- **No** new base classes for services; all domain services extend `RepoService<Entity>`.
- **No** files or folders outside the structure defined in section 1.
- **No** bypassing the global `errorHandler` or duplicating its logic in routes/controllers.
- **No** architectural changes (e.g. new layers, new patterns) unless this contract is updated first.

---

## 12. Reference implementation: Auth module

Use the Auth module as the single source of truth:

- **Route**: `src/routes/auth.route.ts` — Router, AuthController, POST/GET with optional authMiddleware.
- **Controller**: `src/controllers/auth.controller.ts` — register, login, logout, getMe; validator + DTO; service call; ApiResponse.success + ErrorMessages; next(err).
- **Service**: `src/services/repo/auth/auth.service.ts` — extends RepoService<User>, register/login; Ensure.alreadyExists, Ensure.exists; JwtService; no direct getRepository.
- **DTOs**: `src/dto/auth/auth-register.dto.ts`, `src/dto/auth/auth-login.dto.ts` — yup schema + InferType export.
- **Middleware**: `src/middlewares/auth.middleware.ts` — getRepository(User) for current user, Ensure.exists(user, "user"), sets req.currentUser.

Any new module must be consistent with this structure and these patterns.
